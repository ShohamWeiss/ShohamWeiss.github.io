Project Report

Abstract

In this project we developed 3 conditional generative adversarial networks (cGAN) according to the pix2pix methodology. Each model is given as input 256x256 sketched outline images of an anime. The model has the goal of generating an image similar to the colorways of the same anime style. This pix2ix method has been growing in the past decade, and we believe our implementation has industry potential helping animators color frames faster. To evaluate the model results, we crowdsourced ~60 humans for their observations on the model performance. The main feedback has been that some of the models perform impressively well and others less so. Per the survey, the models can struggle with color spilling over edges, and incorrect color choices in certain areas, while performing well at object detection, texturing, and shading.

Introduction

Anime coloring plays an important role in the Anime industry. But coloring the anime is a very tedious and labor-intensive job. Like, if an animation is 1 hour long and uses 24 frames per second. Then, the animator has to color 86,400 frames. If we assume that it takes 5 minutes to color per frame then it will take 7200 hours to color the whole animation.

In recent years the Conditional Adversarial networks (cGAN)  has gained a lot of success in a lot of new image reconstruction tasks. With the help of the cGAN model, we propose to do the work of coloring anime frames in significantly less time. We train a model using machine learning algorithms to color images according to user requirements. The same task without machine learning requires lots of manual effort, time and money.

 

We are developing an app for automating the coloring of anime using our models. Where users can feed input as image to our model through the mobile app’s UI to get the colored image as the result. We have trained our models based on three different anime styles.  So, the user will have an option to select any one of them according to which he wants to color his image.

Methods used

Model Used

cGAN

We decided to use a conditional generative adversarial network (cGAN) as it has been shown to work well for image generation tasks (1). Seen in figure 7, the cGAN model has two sub architectures, the generator and the discriminator. The generator takes in some input, usually noise, and in our case a 256x256 image. With this input, the generator generates a prediction, in our case another 256x256 image. The discriminator takes in as input the generator’s output but sometimes, it takes in as input the ground truth. The discriminator does not know which source the input came from. The dictiminator then tries to predict if its input is the ground truth or a generated image. As the model trains, the discriminator is trained to detect fake images better. Consequently, the generator is trained to generate images that resemble the ground truth better. In our case, the generator learns to generate images with coloring resembling the chosen anime style. A detailed layer by layer diagram of our generator and discriminator models can be seen in figures 9 and 10.

Figure 7: A high level graph of the cGAN. On the left: our input image. On the top: an example generated image. On the bottom: an example ground truth image.

The Generator: Autoencoder

In order to get a generator that will take in an image and generate an image, the generator component of the CGAN has to be a convolutional autoencoder. As seen in figure 8, the convolutional autoencoder takes in an image and uses convolutional layers to encode the image into a latent space. It then uses the latent representation and previous layers to reconstruct the image. This latent space-representation and reconstruction of the image is what allows for the model to learn new ways to construct the image with additional features, such as coloring.

Figure 8: A graph representation of the generative model as an autoencoder. On the left: an example input image, in the middle: the autoencoder, and on the right: the predicted image. This autoencoder is a simplified model, the real autoencoder has convolutional layers.

Data Used

Data Scraping

For each of the anime styles, we downloaded up to 100 episodes of that anime. For each episode, we extracted a frame every 3-5 seconds. Specifically, Attack on Titan: 85 episodes (all seasons + OVA) - 25,216 frames (5 secs/frame), Dragon Ball: 100 episodes (season 1) - 29,602 frames (5 secs/frame), and Fruit Basket: 64 episodes (all seasons) - 30,088 frames (3 secs/frame). These colored frames are our labels for the CGAN model.

Data Preparation

        Using the collected frames, we created a preprocessing script for converting the colored images into grayscale outline sketches. We used opencv’s adaptive threshold image processing capabilities to create the training data as seen in figure 11. First, we convert the images to grayscale. Second, we run adaptive thresholding steps with different thresholds to get a variety of outlines. We then combine the outlines created by the different thresholding in several ways to resemble different sketching and shading styles of the same image. All of the combinations, resulting in 8 images, are used for training.

Figure 11: A diagram of the data scraping and preparation process. The progression is from left to right. For each episode: First, the episode is downloaded. Second, the individual frames are extracted. Third, the images are cropped and resized to 256x256 pixels. Fourth, the images go through processing to create 8 unique grayscale outline sketch representations of the image.

Training

        For every model, we trained and tested on over 1 million images (color frames * 8 styles/frame) and saved a version of the model every 25,000 steps. We first split 80% training and 20% testing for the original frames. The model version that performed the best in the testing dataset will be validated with the real human sketch arts of the same anime (about 50-70 real sketches/anime). We stopped training after saving model 36 (after around 900,000 steps). However we chose model 25 to be the final models for all the anime themes.

Inference

Planned Architecture

We initially planned to use AWS Sagemaker as our main cloud service for training models and running inference on models. With this plan, as seen in figure 12, we also wanted to use either AWS Lambda or AWS EC2 as the API endpoint for the client to use. This api endpoint would do the necessary preprocessing and invoke the inference endpoint in sagemaker. With all of this, it makes sense for everything to be stored in AWS S3. This plan was created while under the impression that the free tier in aws would cover a gpu instance for some training in sagemaker and for some inference. After looking into this, we discovered that the free tier only covers cpu and not gpu resources for training in the free tier.

We decided to shift our design to training on google colab as it has free gpu resources and one of our team members was already paying for the premium version of this service. Then we would transfer the models over to AWS Sagemaker where the inference would take place. This plan turned out to be problematic as we. We trained the models on the latest version of TensorFlow, version 2.8, and were unable to transfer these models into sagemaker as it only supported versions 1.x of TensorFlow.

With this second issue, we decided to shift our design again and keep everything in google colab. This brings us to our final design, described in the next section.

Figure 12: A diagram of our original project architecture.

Final Architecture

        Our final architecture, as seen in figure 13, leverages google drive and google colab cloud services. A google colab notebook is made for each of the three stages of the project, data scraping, model training, and model inference. All the data and models are saved in a shared folder in google drive.

In the first stage of the project, we have data scraping. Here, we download anime episodes from the internet in video form into the notebook. We then process the videos and convert them into 256x256 images. Finally, we save the .jpg image data in the shared drive in google colab to be used as training data.

In the second stage we run the model training. Here, we pull in the prepared training data from the shared drive, and train models using the provided gpu resources in google colab. As the model trains, the model checkpoints are saved in the shared drive. Every 25,000 steps the whole model is saved in .h5 formatting. For any further training, the models are loaded back in from the saved checkpoints in the shared drive.

For the third stage we stand up an inference API service. All of the trained models are loaded in memory of the google colab virtual machine. The same one running the colab notebook. This machine is then run as a flask server with api endpoints for inference. We also set up ngrok to be able to tunnel requests from a globally available IP. With this set up, the mobile app can send an API request to the running notebook. The payload of the request will contain an encoded version of the image to be colored. The notebook decodes the image, converts the image to a model-friendly numpy input, runs the loaded models on the input, and converts the model’s output back to image format. The notebook then responds to the request with each model’s encoded predicted image in json format.

Model Difficulties

Currently our models still cannot detect more than 3 faces in an image, and it cannot color the skin/ clothes naturally either. For example, while we expected the character’s arms to both have a constant skin color (from light pink/yellow to dark brown/gray), the model may put one arm yellow, while the other pink. Color bleedings still exist. Even though we fed the model with 8 different sketch styles, lighting and contrast to make the model more sensitive with different human sketches, the real human sketches are still too varied. Sometimes the model does not detect the lines well. This occurred especially on small images with large white surrounding space.

Discussion

Survey discussion

Overall many agreed that Attack on Titan has more depth/shadings than the other 2, while Dragon Ball gave a wider range of colors and Fruit Basket balanced out with soft/subtle colors and clearer line detections. Both Attack on Titan and Dragon Ball earned the most preferences, while Fruit Basket is a little bit less effective, but still plays a decent part in people’s choices. In terms of overall outputs’ quality, Attack on Titan outweighs the other 2 models with mostly 4 and 5 ratings, while Dragon Ball got the wide range from 1 to 5, and Fruit Basket got mostly 4.

Lessons learned

When we planned our project, we spent the most time training the first model, which was Attack on Titan, thinking that all 3 models will use the same codes with the same data preprocessing and training process and thus will produce the same results. When we got satisfied with Attack on Titan, we implemented the same process with the Dragon Ball model, but we quickly saw that the outputs were not as stable as Attack on Titan. We soon recognized that each model still needed time for tuning, but we already ran out of time when training the third model (Fruit Basket). We somehow believe that the data preprocessing of Fruit Basket (8 sketch styles we chose for Fruit Basket) and the training steps may not be the most optimal, so the Fruit Basket’s output results were less effective than the other 2 models. Our lesson learnt here is that we need to spare more time to expect things to happen, so that we can come back and fix the model if necessary.

Improvements

        As we discussed the models’ weaknesses in the previous section, we suggest some more modification attempts to maximize the output results. There should be some separate training models to recognize human’s parts such as faces, bare hands, bare arms/legs, human’s body, hair as well as common objects such as sky, trees, flowers, buildings. Another option is that we can give color hints on different regions before putting the images into training. After the model generates outputs, we can use another pretrained model to auto add/decrease lightness/contrast of the output images to make them most aesthetically pleasant in the audience's eyes.

        Because our current sketches in the training step were not “real human sketches”, the results are not promising on all sketches. We plan to use the rating system to decide the good/bad outputs generated by the models. With the good color outputs, we can use the user's real sketches to augment our training dataset and continue to train our model (transfer learning), so that the model can gradually learn how to color real human’s sketch arts.

Conclusion

The pix2pix model was used to successfully color fanart of anime in the anime’s coloring style. Training on over 1 million images per model resulted in models that seemed satisfactory according to surveyed members. The models struggled with some edge detection, color choice, and color consistency sometimes noticeably and sometimes not. The models also exceeded our expectations with texture, shading, and object detection capabilities. We believe that with more work, possible preprocessing and post processing, the model could color images as well as a human and better match the colorway of a specific anime.

Possible future work

We noticed that when we fed the models a cropped image, where the model had to guess on to a smaller section of a sketch, the results would improve dramatically. Especially removing white space from the sketches. A possible future direction would be to use a moving window technique to help the model focus on coloring one section at a time and then merge the sections.

The model could use a little more guidance for choosing the correct color. An adjusted version of this model can be made so that the user gives hints of the colors that they want in a region and the model does the shading, gradient and texturing work. This could allow for the model to perform much better.

Lastly, another improvement would be to create a bigger distribution of training data that would account for an even wider variety of differences that come up with sketched images. Maybe even including large white spaces to resemble sketches on paper.
